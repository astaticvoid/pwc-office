# ADR 0007: Service worker with tiered caching for offline support

## Status
Proposed

## Context
The Daily Office is used in contexts with unreliable or no connectivity:
churches (building materials block signal), retreats (remote locations), and
travel. Users expect the app to work offline after an initial visit.

A previous service worker implementation was removed because it cached `app.js`
aggressively, causing stale application code to be served alongside current data
after deploys. The current `web/sw.js` is a kill-switch: it exists solely to
unregister itself and clear residual caches. AGENTS.md records this as a
standing prohibition: "Do not add SW caching back."

This ADR reverses that prohibition. The prior failure mode (stale code +
current data) is a specific caching-strategy error, not an inherent property of
service workers. A correctly-designed SW with version-aware invalidation and
appropriate cache strategies avoids that failure mode while providing offline
support.

## Decision
Implement a **service worker with tiered caching strategies**, differentiated by
resource type:

| Resource type | Strategy | Rationale |
|---|---|---|
| App shell (`index.html`, `app.js`, `render.js`, `office.css`) | Cache-first with manifest check | Delivers instant loads on all visits — including weak/patchy connections. Freshness is verified via `version.json` hash comparison at startup, not via network-first on every request. |
| Core data (`offices.json`, `psalter.json`, `collects.json`, `season_bounds.json`, `fats/saints.json`) | Cache-first with background refresh | Liturgy changes rarely; instant load from cache; background update for next visit. |
| Lectionary data (`/data/lectionary/*`) | Cache-first with background refresh | Monthly rotation; cached files cover the current window. |
| KJV translations (`/data/translations/kjv/*`) | Cache-first | Fixed text, never changes. These are effectively preloaded. |
| External API (NRSVUE: `api.scripture.api.bible`) | Network-only | No caching of scripture API responses. |
| Everything else | Network-first | Default. |

The choice of **cache-first for the app shell** (not network-first) is deliberate.
The stated use case is unreliable connectivity — churches, retreats, travel. A
network-first strategy on weak WiFi makes the user wait for a timeout before
seeing any content. Cache-first delivers the app instantly, then checks for
updates. Freshness is guaranteed by the version manifest, not by per-request
network fetches.

### Cache invalidation
A **version manifest** (`data/version.json`) is generated at build time and
deployed alongside the app in each release directory:

```json
{
  "build": "2026-07-16T120000Z-abc1234",
  "files": {
    "data/offices.json": "sha256-def456...",
    "data/psalter.json": "sha256-abc123...",
    "data/lectionary/2026-07.json": "sha256-ghi789...",
    "data/lectionary/2026-08.json": "sha256-jkl012..."
  }
}
```

On startup, `app.js` fetches `/data/version.json` (network-first — this is the
single request that determines freshness). It compares each file's hash against
the stored hash from the previous session. Only files whose hashes differ are
invalidated and re-fetched. All unchanged files serve from cache instantly.

This is **selective invalidation**, not all-or-nothing. A lectionary rotation
invalidates one monthly file, not the entire KJV Bible.

### Version manifest generation
`version.json` is generated by `make build`. A new tool
(`tools/generate_version_manifest.py`) computes SHA-256 hashes for all data
files in `dist/data/` and writes them to `dist/data/version.json`. This runs as
the final step of the build before the deploy stage. The manifest format is
specified above; the generator script is the sole writer.

### Development mode
During `make serve` (local development with `web/data` as a symlink),
`version.json` is not present. The SW detects the 404 on startup and enters a
pass-through mode: all cache strategies reduce to network-first, no selective
invalidation runs. This ensures local development reflects live file changes
without stale caches. The full caching behavior is exercised in the staging
environment where `version.json` is present.

### Background refresh
Background refresh is triggered on each `fetch` event for cache-first resources:
the cached response is returned immediately; a network fetch is initiated in
parallel only if the `version.json` hash check at startup indicated that the
specific file has changed. Unchanged files serve from cache without a parallel
fetch, preventing redundant network traffic on metered connections.

Fetches that fire before the `version.json` startup check completes are served
from cache unconditionally — the in-memory changed-files set is treated as
empty (nothing invalidated) until the check populates it. This means the very
first page load after SW activation always serves fully cached content; the
freshness check's results take effect on the next navigation.

### Storage quota
Browsers cap SW cache storage. The SW monitors quota usage on activation via
`StorageManager.estimate()` and evicts the least-recently-used lectionary files
if usage exceeds a configurable threshold (default: 80%). Core liturgy data and
KJV translations are never evicted. If the quota API is unavailable (older
browsers), quota management is skipped — the cache may exceed the limit and the
browser will evict entries at its discretion. This is a graceful degradation,
not a correctness failure.

### Offline indicator
A status bar is rendered when the device reports offline (`navigator.onLine` or
the Capacitor Network plugin). The indicator is a DOM element managed by
`app.js` (not `render.js` — it is a shell concern, not a rendering concern).
The banner reads "You are offline. Using cached data." and auto-hides when
connectivity returns.

### Interaction with deploys (ADR 0006)
Each release directory is self-contained: its `version.json` carries the hashes
for its own data files. During CloudFront propagation, users hitting different
edge nodes may receive `version.json` from different release directories — but
the hashes within each file are internally consistent. A user whose shell came
from `vOLD` but whose `version.json` came from `vNEW` will detect hash
mismatches only on files that actually changed between versions, and will
re-fetch only those files. No corruption.

In-flight SPA sessions detect the deploy on their next `version.json` check
(periodic, on navigation) and can prompt a reload if the build hash changed.

## Consequences

### Positive
- The app works offline after first visit. All liturgy data, lectionary, and
  KJV text are cached.
- Cache-first shell strategy provides instant loads even on unreliable
  connections — the user never waits for a network timeout.
- Selective invalidation via per-file hashes means a lectionary rotation
  invalidates one file, not the entire cache.
- Version-manifest invalidation is deterministic: the app knows exactly when
  its cached data is stale, with no implicit cache-busting heuristics.

### Negative
- Service worker complexity: ~120 lines including cache strategies, version
  manifest checking, selective invalidation, and quota management.
- Two named caches to manage: one for the application shell and one for data
  files, with different eviction policies.
- Playwright E2E tests must include an offline scenario to verify cached
  content renders correctly without network access.

### Notes
- The Capacitor Network plugin (already integrated) provides the connectivity
  signal. The SW `fetch` handler provides the caching — the two are independent,
  not coupled.
- The AGENTS.md prohibition against SW caching is explicitly reversed by this
  ADR. The prior failure was a caching-strategy error (network-only for data,
  cache-first for shell) that caused stale-code/current-data mismatches. The
  new design's cache-first shell + version-manifest invalidation does not have
  that failure mode.
